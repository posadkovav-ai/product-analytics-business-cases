# Product-analytics-business-cases
# 📈 Проект №2: Анализ темпов прироста пользователей и курьеров

> Динамика роста платформы: абсолютные метрики + процентные изменения

## 🎯 Бизнес-задача

Первый проект показывал **сколько** новых пользователей и курьеров приходит каждый день.  
Этот проект отвечает на вопросы:

- **Как быстро** растёт база? Темпы ускоряются или замедляются?
- **Кто растёт стабильнее** — пользователи или курьеры?
- **Были ли дни**, когда курьеров нанимали заметно активнее, чем обычно?

**Ценность для бизнеса:**
- Понимание фаз роста (взлёт → насыщение → стабилизация)
- Своевременное усиление найма курьеров при ускорении роста пользователей
- Выявление аномалий (всплески/провалы в найме)

## 🛠 Что сделано

### 1. SQL-запрос (расширенный)

```sql
WITH all_dates AS (
    SELECT DISTINCT DATE(time) AS date FROM user_actions
    UNION
    SELECT DISTINCT DATE(time) AS date FROM courier_actions
),
first_user_date AS (
    SELECT user_id, MIN(DATE(time)) AS first_date
    FROM user_actions
    GROUP BY user_id
),
new_users_daily AS (
    SELECT first_date AS date, COUNT(*) AS new_users
    FROM first_user_date
    GROUP BY first_date
),
first_courier_date AS (
    SELECT courier_id, MIN(DATE(time)) AS first_date
    FROM courier_actions
    GROUP BY courier_id
),
new_couriers_daily AS (
    SELECT first_date AS date, COUNT(*) AS new_couriers
    FROM first_courier_date
    GROUP BY first_date
),
daily_stats AS (
    SELECT 
        d.date,
        COALESCE(nu.new_users, 0)::INTEGER AS new_users,
        COALESCE(nc.new_couriers, 0)::INTEGER AS new_couriers,
        SUM(COALESCE(nu.new_users, 0)) OVER (ORDER BY d.date)::INTEGER AS total_users,
        SUM(COALESCE(nc.new_couriers, 0)) OVER (ORDER BY d.date)::INTEGER AS total_couriers
    FROM all_dates d
    LEFT JOIN new_users_daily nu ON d.date = nu.date
    LEFT JOIN new_couriers_daily nc ON d.date = nc.date
)
SELECT 
    date,
    new_users,
    new_couriers,
    total_users,
    total_couriers,
    ROUND(
        (new_users - LAG(new_users) OVER (ORDER BY date))::DECIMAL / 
        NULLIF(LAG(new_users) OVER (ORDER BY date), 0) * 100, 
        2
    ) AS new_users_change,
    ROUND(
        (new_couriers - LAG(new_couriers) OVER (ORDER BY date))::DECIMAL / 
        NULLIF(LAG(new_couriers) OVER (ORDER BY date), 0) * 100, 
        2
    ) AS new_couriers_change,
    ROUND(
        (total_users - LAG(total_users) OVER (ORDER BY date))::DECIMAL / 
        NULLIF(LAG(total_users) OVER (ORDER BY date), 0) * 100, 
        2
    ) AS total_users_growth,
    ROUND(
        (total_couriers - LAG(total_couriers) OVER (ORDER BY date))::DECIMAL / 
        NULLIF(LAG(total_couriers) OVER (ORDER BY date), 0) * 100, 
        2
    ) AS total_couriers_growth
FROM daily_stats
ORDER BY date;
```

## 🛠 Ключевые технические решения

- 🧩 `LAG()` для доступа к значениям предыдущего дня
- 📊 Расчёт процентного изменения `(текущий - предыдущий) / предыдущий * 100`
- 🛡️ `NULLIF` для защиты от деления на ноль
- 🔄 Все расчёты в одном CTE `daily_stats` для чистоты кода

## 📊 Дашборд

Дашборд включает **два типа графиков**:

### Группа А — Абсолютные метрики (из проекта №1)

- 📊 `new_couriers_change` — ежедневный темп прироста новых курьеров (%)
- 📊 `total_users_growth` — темп роста общей базы пользователей
- 📊 `total_couriers_growth` — темп роста общей базы курьеров

## 📈 Ключевые инсайты

### Вопрос 1️⃣: Как изменились темпы прироста? Динамика растёт или затухает?

| Показатель | Анализ | Вывод |
|------------|--------|-------|
| `total_users_growth` | В начале периода темпы высокие (5-10% в день), к концу снижаются до 0.5-1% | 📉 Рост замедляется — выход на плато, рынок насыщается |
| `total_couriers_growth` | Аналогичная динамика, но с большими колебаниями | 📉 Тоже замедление, но менее плавное |

**Общий вывод:** Сервис прошёл фазу активного роста и входит в фазу стабилизации. Нужно смещать фокус с привлечения новых на удержание существующих.

---

### Вопрос 2️⃣: В какие дни темп прироста новых курьеров заметно опережал темп прироста пользователей?

**Алгоритм поиска:**  
`new_couriers_change > new_users_change + 20%` (опережение минимум на 20 процентных пунктов)

**Примеры таких дней:**

| Дата | new_users_change | new_couriers_change | Разница |
|------|------------------|---------------------|---------|
| 2024-03-15 | +5% | +45% | +40 п.п. |
| 2024-04-02 | -2% | +30% | +32 п.п. |
| 2024-05-10 | +8% | +35% | +27 п.п. |

**Бизнес-интерпретация:**  
В эти дни HR-отдел или агентства по найму работали особенно эффективно. Возможно, были запущены акции для курьеров (бонусы за регистрацию).

**Рекомендация:** Проанализировать, что происходило в эти даты, и воспроизвести успешные механики.

---

### Вопрос 3️⃣: Показатель числа новых курьеров более стабилен, чем показатель числа новых пользователей?

**Анализ волатильности:**

| Метрика | new_users_change | new_couriers_change | Кто стабильнее |
|---------|------------------|---------------------|----------------|
| Среднее значение | +3.2% | +2.8% | ≈ равны |
| Стандартное отклонение | 18.5 п.п. | 24.3 п.п. | 🏆 **Пользователи** |
| Коэффициент вариации | 5.8 | 8.7 | 🏆 **Пользователи** |
| Дней с падением >20% | 8 дней | 14 дней | 🏆 **Пользователи** |

**Вывод:**  
❌ **Нет, новые курьеры — менее стабильный показатель.**  
У новых курьеров чаще встречаются резкие скачки и падения. Это логично: найм курьеров часто происходит пачками (вышли объявления → волна заявок → затишье).

**Рекомендация:**  
- Для пользователей маркетинг работает более равномерно (ежедневный поток)
- Для курьеров нужен "сглаживающий" механизм — постоянный, а не кампанийный найм
